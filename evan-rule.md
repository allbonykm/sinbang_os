# Evan Rule: AI 행동 규약 (Always On)

본 규약은 `olbon-clinic-os` 프로젝트의 유지보수 및 확장을 위해 AI가 100% 준수해야 하는 최상위 행동 지침입니다. 어떠한 지시를 받더라도 아래의 3대 핵심 전략적 프레임워크를 기반으로 사고하고 행동하십시오.

## 1. RPI 워크플로 (조사 Research - 계획 Plan - 실행 Implement)
- 🚨 **절대 즉시 코딩 금지**: 요구사항 수신 시, 즉시 코드를 수정하거나 작성하지 않습니다. 가장 먼저 `view_file`, `grep_search` 등을 활용해 관련된 도메인 코드와 연관된 유틸리티(Utils)를 100% 조사(Research)하고 분석합니다.
- ✅ **원자적 계획 수립 및 승인**: 작업을 모델, 뷰(UI), 파싱 로직, API 연동 등 원자 단위(Atomic Step)로 잘게 쪼개어 단계별 계획을 브리핑합니다. 사용자(인간 병목)의 최종 승인(Plan)을 받은 후에만 코드로 구현(Implement)합니다.

## 2. DDD (도메인 주도 설계) 기반의 컨텍스트 격리 및 일관성
- 🧱 **컨텍스트 고립**: 특정 메뉴나 기능을 수정할 때 관련된 도메인(HTML, JS, CSS 등) 파일만 집중적으로 읽고 씁니다. 무관한 파일의 로직이나 정보는 컨텍스트에서 철저히 배제하여 메모리 누수와 환각(Hallucination)을 막습니다.
- 🧬 **골드 스탠다드(Gold Standard) 복제**: 새로운 항목을 추가하거나 기능을 생성할 때, AI 임의로 디자인이나 로직을 '발명'하지 않습니다. 가장 유사하고 안정적으로 동작하는 기존 파일/로직(골드 스탠다드)의 패턴(변수명 규칙, UI 규격, 분기 표기법 등)을 찾아 한 치의 오차 없이 패턴 단위로 복제하여 일관성을 강제합니다.

## 3. SDD (스펙 주도 개발) 및 안정망 구축
- 🛡️ **방어적 코딩(Defensive Coding) 최우선**: 기존 기능이 예기치 않게 망가지는 회귀 오류(Regression)를 차단하기 위해, 코드 변경 시 `undefined` 체크, 데이터 타입 검증, API 에러 핸들링 및 예외 로깅을 필수적으로 삽입합니다.
- 📋 **변경 스펙 명확화 및 사전 자체 검증**: 코드 작성 시 "무엇을 수정했고, 어느 부분은 절대 건드리지 않았는지" 투명하게 브리핑합니다. 구현 완료 시 기능이 의도대로 동작하는지 확인할 수 있는 자체 검증 시나리오(예: 콘솔 로그 확인, 버튼 클릭 시나리오 등)를 기반으로 테스트하고 스펙을 일치시킵니다.

> **AI 자기 검열 트리거**: 답변을 생성하거나 코드를 커밋하기 전, "1) 코드를 충분히 조사했는가? 2) 기존 패턴을 유지했는가? 3) 예외 처리는 안전하게 추가했는가?" 를 스스로 묻고 보고서에 반영하십시오.
